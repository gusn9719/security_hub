> 전체 통합본: [../상세 파악 가이드.md](../상세%20파악%20가이드.md) (섹션 4~5)

## 투표 시스템

모듈: `backend/database/vote_service.py`

`save_vote()`: vote가 `safe/danger/spam`이면 `INSERT OR IGNORE`로
부분 UNIQUE 인덱스 슬롯을 점유. `unsure`는 **DB에 저장하지 않음**(슬롯
미점유 — 나중에 진짜 의견으로 재투표할 수 있게 비워둠).

```sql
UNIQUE (device_uuid, registered_domain)
  WHERE vote IN ('safe', 'danger', 'spam')   -- unsure 제외
```

`get_vote_counts()`: `user_id IS NULL`이면 `anon`, 아니면 `user` 버킷으로
분리 집계(`anon_safe/anon_danger/anon_spam`, `user_safe/user_danger/
user_spam`, 합계 `safe/danger/spam/total`).

## 어그로 방어 5중

| Layer | 구현 | 위치 | 효과 |
|---|---|---|---|
| 1. DB | 부분 UNIQUE 인덱스 | `vote_service.py` + 스키마 | 1기기 1표(도메인당) |
| 2. Logic | 우세방향 가드 | `heuristic_scorer.py` | danger>safe 등 명확한 우세일 때만 시그널 발동, 동률이면 무발동 |
| 3. UX | 체류시간 | `browse_service.py` `_auto_terminate` 5분 | **불완전** — 강제 체류 아니라 자동정리 타임아웃일 뿐. 30초 최소체류 강제는 미구현 |
| 4. Score | 점수상한 | `heuristic_scorer.py` | `prior_danger_vote_high` 최대 +35, 단독 DANGER 불가(70 미달) |
| 5. Auth | 카카오 인증 | `auth.py`/`user_service.py` | 가입자 1명=익명 3~3.3명 권위(임계값 anon≥10 vs user≥3) |

**Layer 3는 정직하게 한계로 인정해야 함** — CLAUDE.md에도 "UX 체류시간"
이라 써있지만 실제로는 5분 타임아웃(컨테이너 자동 정리)일 뿐, 30초 미만
체류 시 투표를 막는 강제 로직은 없다. 질문 들어오면 "현재는 소프트
가드(Layer 1·2·4·5)만으로 충분히 비용 비대칭을 만들고 있고, 강제 체류
타이머는 졸업작품 범위에서 보류했다"고 답하는 게 정확함.

### 투표→재분석 승급 엔드투엔드 실증 (2026-06-21)

상세: [../sandbox_eval.md](../sandbox_eval.md) §6. 테스트 URL
`http://8.8.8.8/promo`(ip_in_url 단일 시그널, baseline 35점)를
`analyze()` → **suspicious** 확인 → device_uuid 10개로 danger 투표
적재 → 재분석 → **danger**로 승급(35+35=70). `prior_danger_vote_high`가
실제 DB 투표를 거쳐 살아있는 파이프라인에서 정확히 반영됨을 처음으로
실데이터 확인(`backend/tests/run_vote_escalation_test.py`). 부수 발견:
`AnalyzeResponse`가 점수 숫자/triggered를 노출하지 않아 "왜 올랐는지"는
카드 텍스트로만 추론 가능(투명성 공백, 보류 항목).

## 인증 — 카카오 로그인 + JWT (AUTH-01~03)

### 백엔드 흐름

```
Flutter (kakao_flutter_sdk_user, talk→web 폴백)
  → POST /auth/kakao { access_token }
  → kakao_service.fetch_user_info()  — kapi.kakao.com/v2/user/me 직접 검증, 5초 타임아웃
  → user_service.upsert_by_kakao_id() — 신규면 INSERT, 기존이면 nickname/email/last_login_at UPDATE
  → jwt_service.issue_token(user_id) — HS256, sub=user_id, exp=720시간(30일)
  → 응답 { access_token, user: {id, kakao_id, nickname, ...} }
```

- **JWT 시크릿 32자 미만 거부**(RFC 7518 §3.2) — `jwt_service.py`가
  앱 시작 시점에 검증, 짧으면 `ValueError`.
- **PII 마스킹**: `user_service._mask_kakao_id()`가 로그에 `1234******`
  형태로만 남김. kapi 응답 body 자체도 로그에 안 남김.
- `OptionalAuthMiddleware`: 토큰 없으면 익명 통과(`request.state.user_id =
  None`), **무효 토큰(만료/서명오류)은 무조건 401** — silent pass-through로
  두면 클라이언트가 죽은 토큰을 계속 들고 있어서 익명 가중치로 계속 동작하게
  되므로, 명시적으로 401을 줘서 클라이언트가 즉시 폐기하도록 강제.

### 프론트엔드 흐름

`AuthService`(`frontend/lib/services/auth_service.dart`): talk 로그인
실패 시(사용자 취소가 아닌 진짜 오류) web 로그인으로 폴백 → 백엔드 교환 →
JWT+프로필을 SharedPreferences에 영속화. `ApiService._headers()`가 매
요청마다 JWT 존재 시 자동으로 `Authorization: Bearer` 부착. 401 수신 시
`AuthService.clearLocalAuth()`로 즉시 폐기 후 익명 모드 전환(자동 재로그인은
시도 안 함 — 카카오 SDK가 사용자 인터랙션을 요구하기 때문).

`LoginScreen`: 카카오 로그인 버튼(공식 색상 #FEE500) + "비회원으로 시작"
버튼 — 비회원 선택도 `markAnonymous()`로 영속화해 다음 실행에 또 묻지 않음.

### 가입자/익명 임계값 분리 (DC-45)

```
prior_*_vote_high : anon_X >= 10  OR  user_X >= 3
prior_*_vote_low  : anon_X >= 3   OR  user_X >= 1
```

우세 방향 가드(`danger > safe` 등)는 합계 기준 그대로, 분리는 임계값
비교에만 적용. "카카오 가입 = 자연인 1명 = 익명 100개 중 카카오는 휴대폰
회선 100개+SMS 인증 100회 필요" 비용 비대칭이 핵심 근거.
