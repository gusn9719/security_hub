> 전체 통합본: [../상세 파악 가이드.md](../상세%20파악%20가이드.md) (섹션 2)

오케스트레이터: [`backend/services/analysis_service.py`](../../backend/services/analysis_service.py)

Early Return 구조. 동기 블로킹 I/O(SQLite, WHOIS, 단축URL 해제)는 모두
`asyncio.to_thread()`로 분리해 이벤트 루프를 막지 않는다(DC-42).
분석 이력은 `BackgroundTasks`로 응답 후 비동기 저장(DAT-06) — 응답 지연 없음.

## 0~1단계 — URL 추출 & 위험 스킴 검사

| 단계 | 모듈 | 동작 |
|---|---|---|
| 0 | `blacklist_service.extract_urls()` | 프로토콜 URL 정규식 + 프로토콜 없는 베어 도메인(`KNOWN_TLDS` 화이트리스트로 오탐 방지) 추출. URL 없으면 즉시 SUSPICIOUS |
| 1 | `url_validator.check_dangerous_scheme()` | `javascript:/data:/vbscript:/file:/blob:` → 즉시 DANGER (ANL-00) |

**한계 (미해결 P0-2)**: `check_dangerous_scheme()`이 `unquote()`나 `strip()`을
하지 않는다. `j%61vascript:alert(1)`(퍼센트 인코딩) 이나 앞에 공백이 붙은
` javascript:alert(1)`은 `urlparse().scheme`이 빈 문자열이 되어 이 단계를
그냥 통과한다. 다만 7단계 `double_encoding` 시그널(+15점)만으로는 DANGER
임계값(70)에 못 미쳐 SUSPICIOUS로 폴백된다 — 위험하지만 안전망 안에 있다.

## 2단계 — 단축 URL 해제 + SSRF 방어

모듈: `backend/services/url_expander.py`

- `is_short_url()`: `SHORT_URL_DOMAINS`(bit.ly, forms.gle, goo.gl, naver.me,
  han.gl, url.kr, me2.do, t.co, fb.me, alie.kr, ph.link 등 17개+) 포함 여부.
- `expand_url()`: HEAD 요청(`allow_redirects=False`), 최대 **3-hop**, 루프
  탐지(visited set), Android User-Agent(일부 단축서비스 봇 차단 우회용),
  타임아웃 5초. 실패 시 원본 URL 유지(SUSPICIOUS 폴백 안전망).
- **SSRF 방어**: `_is_safe_to_request()`가 매 hop 전에 `is_private_ip()`로
  사설/루프백/링크로컬 IP 목적지를 차단.

**최근 수정 (P1-1, DC-49, 2026-06-04 발견)**: 신규 함수 `_normalize_ip_host()`
추가. 이전에는 `2130706433`(10진수), `0x7f000001`(16진수), `0177.0.0.1`
(8진수 옥텟) 같은 비표준 IP 표기를 `ipaddress.ip_address()`가 거부해
`is_private_ip()`가 `False`를 반환 → SSRF 우회가 가능했다. 지금은 정규화
후 검사하도록 고쳐 세 가지 우회 패턴 모두 `127.0.0.1`로 정확히 정규화된다.

**미해결 한계**:
1. **DNS rebinding 미방어** — URL 파싱 단계에서 별도 DNS 조회를 하지 않고
   `requests` 내부 동작에 의존. 1차 요청에 정상 IP를 응답하고 2차에 사설
   IP로 바꾸는 공격은 막지 못함(경량 설계 트레이드오프, 해결 안 함).
2. **3-hop 초과 우회 (미해결 P1-2)** — 4단계 이상 리다이렉트 체인이면 3번째
   hop URL을 "최종"으로 잘못 반환한다. 예: `bit.ly→tinyurl→is.gd→실제피싱`
   체인이면 `is.gd` 단계에서 멈춰 실제 피싱 도메인을 분석하지 못함.

## 3a/3b단계 — 블랙리스트 3단계 매칭

모듈: `backend/database/blacklist_service.py`

`check_blacklist(urls, hash_only=False)` — **url_hash → domain →
registered_domain** 순서로 매칭(우선순위 = 정확도순):

```
1순위 url_hash 정확일치   — SHA256(이중디코딩+소문자+trailing슬래시 제거 후 URL)
2순위 domain 일치          — hash_only=True면 스킵
3순위 registered_domain 일치 — hash_only=True면 스킵
```

`url_hash` 정규화: `unquote(unquote(url))`(이중 디코딩, OWASP 이중인코딩
우회 방지) → 소문자 → trailing slash 제거 → SHA256.
`load_ctas_csv.py`도 동일 정규화 로직을 공유해야 함(불일치 시 매칭 실패).

**파이프라인에서 3a/3b로 분리한 이유 (DC-46)**: 화이트리스트 보호 *전에*
정확한 피싱 경로(url_hash)는 먼저 막아야 하지만, 도메인 레벨 매칭은
화이트리스트를 통과한 뒤에만 적용해야 단축 URL 서비스가 통째로 막히지 않는다.
그래서 `analysis_service.py`가 `check_blacklist(hash_only=True)`를
화이트리스트 단계 *앞*에서, `check_blacklist(hash_only=False)`를 화이트리스트
단계 *뒤*에서 두 번 호출한다.

**최근 수정 (P0-1, DC-46, 2026-06-04 발견 — 가장 임팩트 큰 버그)**:
C-TAS 블랙리스트에 `forms.gle`, `bit.ly` 등 단축 URL 서비스 도메인 자체가
231건 등재돼 있어서, 2/3순위 도메인 매칭이 그 도메인을 통째로 차단했다.
실제 사례: `"아차사고 공모전 오늘까지 접수 https://forms.gle/aLkBXdQ4..."`
같은 정상 구글폼 공모전 안내가 DANGER로 오판됨. 수정: `_SHORT_URL_PROVIDERS`
frozenset(`forms.gle, goo.gl, bit.ly, naver.me, me2.do, han.gl, url.kr, t.co,
fb.me` 등)을 추가해 이 도메인들은 **url_hash 정확 매칭만** 적용하도록 제한.
정상 경로는 통과, 실제 등재된 피싱 경로는 계속 차단.

**한계**: `_SHORT_URL_PROVIDERS`(blacklist_service.py), `SHORT_URL_DOMAINS`
(url_expander.py) 두 곳을 항상 동기화해야 함 — 신규 단축서비스 추가 시
한쪽만 고치면 누락 위험.

## 4단계 — 화이트리스트 + Open Redirect

모듈: `backend/database/whitelist_service.py`

- 매칭 모드: **exact**(`tistory.com` ≠ `evil.tistory.com`), **suffix**
  (`naver.com` 히트 → `login.naver.com`도 포함). `pattern` 모드는 **DC-001로
  제거됨**(`.go.kr` 같은 패턴이 임의 하위도메인을 다 허용해서 오탐 위험 —
  기관별로 exact/suffix 개별 등록으로 전환).
- `_get_suspicious_categories()`: query string에서 `&` 경계로 정확히 파라미터
  매칭(`image_url=` 이 `url=` 로 오탐되지 않게). 카테고리: 리다이렉트
  (`redirect=/goto=/url=` 등), 자격증명(`token=/passwd=` 등), 파일다운로드,
  로그인우회.
- 화이트리스트 히트 + Open Redirect 파라미터 감지 → **DANGER가 아니라
  SUSPICIOUS** (오탐 가능성 고려한 보수적 처리).

**SAFE로 가는 유일한 경로**: 화이트리스트 히트 AND Open Redirect 없음.
이 외에는 무조건 SUSPICIOUS 이상.

## 5~6단계 — 도메인 평판 (WHOIS/SSL) + 캐시

모듈: `domain_reputation_service.py` + `reputation_cache_service.py`

- WHOIS 등록일 → `domain_age_days` → 30일 이내면 `new_domain` 플래그.
- SSL 인증서 발급일 → `ssl_issued_days` → 7일 이내면 신선한 SSL.
- 둘 다 신규면 `fresh_infrastructure`(피싱 캠페인 단기 인프라 패턴).
- IP 주소·`SKIP_WHOIS_TLDS`(.local/.test 등)는 조회 자체를 스킵.
- **캐시 TTL 7일** — 같은 등록 도메인 재조회 방지(WHOIS 레이트리밋 완화).
- 모든 외부 조회는 실패해도 `None` 처리(graceful) — SUSPICIOUS 폴백 안전망
  안에서 동작, 서비스 중단 없음.

**한계**: python-whois 응답 형식이 레지스트라별로 들쭉날쭉(날짜 포맷 5종
지원해도 못 잡는 케이스 존재), ccTLD 레이트리밋 가능성, HTTP→HTTPS 실제
리다이렉트 여부는 정적 분석이라 확인 불가.

**설계 핵심 — 왜 임계값 70인가**: 가장 강한 단일 시그널(`sandbox_danger_score`)
이 40점이라, 70점에 도달하려면 항상 다른 시그널이 최소 1개 더 결합돼야 한다.
즉 **DANGER 판정은 절대 "단일 근거"로 나오지 않는다** — 설명 가능성과 오탐
방어를 동시에 잡기 위한 설계.

투표 시그널은 익명/가입자 임계값이 분리(DC-45) — 가입자 1명은 휴대폰 인증을
거쳤으므로 익명 3~3.3명 권위로 환산.

**성능평가 체계 (2026-06-21 실측) — 5단 구조**

휴리스틱은 안전/의심/위험 판정의 한 단계일 뿐이다. "시스템 정확도"와
"휴리스틱 컴포넌트 내부 동작"을 명확히 나눠서 본다(이전엔 섞어서 두 번
틀린 결론을 냈었음 — 아래 참조).

- **(a) 시스템 레벨 3분류 정확도(헤드라인, [../system_accuracy_eval.md](../system_accuracy_eval.md))**:
  `analyze()` 전체 파이프라인 실제 호출. 기존셋(508건) **정확도 100%**,
  독립 holdout(250건) danger 탐지율 **100%**/safe FPR **0%**, SRS 목표
  (≥95%) **PASS**.
- **(b) 휴리스틱 컴포넌트 단위 분석([../heuristic_eval.md](../heuristic_eval.md))**:
  `score_url()` 단독 호출(블랙/화이트 우회) — 시스템 정확도가 아니라
  점수분포(danger군 70~130 vs suspicious/safe군 거의 0, **이산적/
  이중모드**)·threshold 민감도·시그널 발화빈도만 본다.
  `test_heuristic_scorer.py`(25종 전부 합성입력, **58개 PASS**)가 위에서
  데이터로 못 검증한 10개 사각지대 시그널(도메인평판 3+투표 6+sandbox 1)
  을 보완.
- **(c) 투표→위험 승급 실증**: SUSPICIOUS(35점)→danger투표10건→**DANGER
  (70점) 승급** 확인(`run_vote_escalation_test.py`).
- **(d) 투표→안전 톤완화 실증(승격 아님)**: SUSPICIOUS(35점)→safe투표10건
  →**점수만 20점으로 감소, status는 suspicious 유지**(`run_safe_tone_down_test.py`)
  — DC-06은 투표 누적과 무관하게 항상 지켜짐.
- **(e) 7-B 샌드박스 실데이터**: [../sandbox_eval.md](../sandbox_eval.md) 참조.

**정정 기록**: 1차 시도에서는 `score_url()` 단독 결과로 시스템 정확도를
재려고 해서 "suspicious Recall=0.000"이라고 두 번 잘못 보고했다(DC-06
정책 보정을 빼먹은 채 raw 내부 판정만 정답과 비교). 진짜 시스템 정확도는
기존 스크립트(`run_performance_eval.py`/`run_holdout_eval.py`)로 다시
측정했고(위 (a)), 휴리스틱 단독 분석은 "컴포넌트 내부 동작"으로 명확히
재라벨링했다(위 (b)).

## 7단계-보조 — 타이포스쿼팅 (`domain_similarity.py`)

순수 파이썬 Levenshtein 동적계획법(외부 라이브러리 미사용, O(len(a)×len(b)),
1차원 롤링 배열로 메모리 절약). `detect_typosquat(registered_domain,
max_distance=2)`:
- 정확 일치면 미발동(정상). 표적 도메인 35개(포털/금융/커머스/글로벌/정부
  — `naver.com, kakao.com, toss.im, hometax.go.kr, coupang.com, paypal.com` 등)
  와 거리 계산.
- 길이 차이가 `max_distance`보다 크면 가지치기(스킵), 길이 < 6인 표적은
  제외(예: `sk.com`은 짧아서 거리 1로도 무관한 도메인과 오탐 가능 — 안전장치).
- 거리 1 발견 시 조기 종료(더 짧은 거리는 있을 수 없음).

**한계**: 표적 도메인 35개로 제한(신규 핀테크 등 미포함), `naver-event.com`
같은 부분 변형은 거리 2 이내면 잡히지만 TLD 변형(`naver.co.kr` vs
`naver.com`, 거리 5)은 못 잡음.

## 8단계 — 설명 카드 (`explanation_service.py`)

`EXPLANATION_DICT`(30개+ 카드, `{icon, title, desc}`)에서 발화된 시그널을
점수 높은 순으로 매핑. **Gemini가 즉석 생성하지 않음**(DC-25, 할루시네이션
방지). `build_safe_cards`/`build_blacklist_cards`(category별 힌트: 공공기관/
택배/금융)도 동일하게 딕셔너리 기반.
