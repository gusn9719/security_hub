## 알려진 한계 총정리

| 영역 | 한계 | 상태 |
|---|---|---|
| 1단계 위험스킴 | 인코딩(`%61`)/공백 우회 (P0-2) | 미해결 — double_encoding 시그널이 안전망 |
| 2단계 URL해제 | DNS rebinding 미방어 | 미해결, 해결 안 하기로 함(경량설계) |

| 7단계 휴리스틱 | `domain_similarity` 16자+ 도메인도 거리2 단일적용(Q-20) | 미해결, 정밀화 보류 |

## 성능평가 체계 5단 구조 (2026-06-21 실측, 시행착오 2회 기록)

휴리스틱은 안전/의심/위험 판정의 한 단계일 뿐이라 "시스템 정확도"와
"휴리스틱 컴포넌트 내부 동작"을 명확히 나눴다 — 처음엔 이걸 섞어서
두 번 틀린 결론을 냈었다(아래 정정 기록).

**(a) 시스템 레벨 3분류 정확도(헤드라인)** —
전체 내용: [docs/system_accuracy_eval.md](../system_accuracy_eval.md).
`AnalysisService.analyze()` 전체 파이프라인을 실제로 호출(기존에 이미
있었지만 한 번도 실행 안 했던 `run_performance_eval.py`/`run_holdout_eval.py`
를 이번에 실행):
- 기존 테스트셋(508건): **정확도 100.0%**, 3분류 confusion matrix 전부 대각선
- 독립 holdout(250건, 튜닝에 안 쓰인 데이터): danger 탐지율 **100.0%**
  (200/200), safe FPR **0.0%**(0/50), SRS 목표(≥95%) **PASS**
- (참고) `run_combined_eval.py`는 holdout 파일 로딩에 버그가 있어 0건으로
  나옴 — `run_holdout_eval.py`를 별도 실행해 우회(수정은 범위 밖)

**(b) 휴리스틱 컴포넌트 단위 분석** —
전체 내용: [docs/heuristic_eval.md](../heuristic_eval.md). `score_url()`
단독 호출(블랙/화이트 우회) — **표기 정정**: 시그널은 `len(_WEIGHTS)`로
확인한 결과 **25종**(CLAUDE.md의 "23종"은 stale 카운트, 이번에 정정).
`test_heuristic_scorer.py`(25종 전부 합성입력, **58개 PASS**)가 도메인평판
3+투표 6+sandbox 1 = 10개 사각지대 시그널을 보완. 점수분포는 danger군
70~130 vs suspicious/safe군 거의 0(이산적/이중모드).

**(c) 투표→위험 승급 실증**: SUSPICIOUS(35점)→danger투표10건→**DANGER
(70점) 승급**(`run_vote_escalation_test.py`).

**(d) 투표→안전 톤완화 실증(승격 아님, 신규)**: SUSPICIOUS(35점)→safe투표
10건→**점수만 20점으로 감소, status는 suspicious 유지**
(`run_safe_tone_down_test.py`) — "안전쪽에도 보수적으로 반영되는가"의
정확한 답: 경고 강도는 줄지만 SAFE 승격은 DC-06으로 항상 막힘.

**(e) 7-B 샌드박스 실데이터**: 아래 절 참조.

**정정 기록(2회)**: 1차는 `score_url()` raw 판정만으로 "suspicious
Recall=0.000"이라 보고(DC-06 보정 누락). 2차는 "effective verdict"로
고쳤지만 이것도 실제 블랙/화이트리스트를 안 거치는 근사치라 시스템
정확도를 대체할 수 없었다. 3차(현재)부터 (a)를 헤드라인으로, (b)는
순수 컴포넌트 분석으로 명확히 분리.

## 7-B 샌드박스 실데이터 신뢰성 평가 (2026-06-21 실측)

전체 내용: [docs/sandbox_eval.md](../sandbox_eval.md). 핵심만:

- 신규 산출물: `backend/tests/run_sandbox_eval.py` — `run_auto_test()`를
  직접 호출(HTTP 계층 우회), 실제 Docker+Playwright로 C-TAS 등재 위험 URL
  10건 + 화이트리스트 공식 도메인 10건을 라이브 방문(동시성 3,
  `_AUTO_SEM` 한계와 동일).
- **죽은 링크**: C-TAS 10건 중 9건이 DNS실패/타임아웃 — 스미싱 인프라의
  빠른 폐기 주기를 보여줌(결함 아님). 살아있던 1건은 점수 0.
- **안전 사이트 오탐 22.2%(2/9)**: `naver.com/security`→`external_form_action`
  (+40), `cjlogistics.com`→리다이렉트 14회로 `redirect_count>=3`(+20).
  둘 다 DANGER(70) 미달.
- 5가지 기준 중 2개(`external_form_action`/`redirect_count>=3`)가 **실제
  공개 사이트에서 자연 발화** — mock이 아닌 실증.
- **발견**: 자체 5기준 테스트 페이지(`backend/static/test_phishing/`)가
  있지만 SSRF 방어로 `host.docker.internal`이 `0.0.0.0`에 막혀 있어
  로컬에서는 못 씀(공개 노출 필요) — 의도된 보안 설계.
- **경미한 발견**: 평가 후 격리 네트워크 1개가 정리 안 되고 남음
  (`container.stop()` 직후 `network.remove()` 호출 시 Docker 엔드포인트
  분리 경합 추정, 컨테이너 자체는 정상 정리됨) — 수정은 범위 밖, 기록만.

전체 내용: [docs/qa_benchmark.md](../qa_benchmark.md). 핵심만:

**전체 14개 항목 PASS 14 / FAIL 0.**

**측정 중 발견한 사실**: 기존 `backend/tests/run_qa_benchmark.py`를 그대로
실행하면 거의 전 항목이 FAIL/측정불가로 나온다. 원인은 서버가 아니라
스크립트 자체의 설계 결함 — `run_perf_whitelist()` 한 함수가 워밍업 10회 +
측정 20회 = 30회를 같은 IP·같은 엔드포인트(`/analyze`)로 연속 호출하는데,
`/analyze`의 레이트리밋이 10회/60초라서 워밍업 단계에서 이미 쿼터를 다
써버린다. 그래서 이전 `qa_benchmark.md`가 항상 "측정 대기" 상태였던 것으로
보인다. 이번엔 카테고리 사이 65초 대기 + 호출량을 쿼터 이하로 낮춘 경량
스크립트로 재측정해 실제 수치를 얻었다. (이건 역설적으로 레이트리밋이
설계대로 엄격하게 작동한다는 증거이기도 함 — 발표에서 좋은 소재.)
