# 발표용 정량 평가 요약 (2026-06-22 발표)

심사 지적 3건(휴리스틱 캘리브레이션·서버 가용성·샌드박스 격리 안전성)에 대한 정량 평가 + 기존 URL 분석(휴리스틱/파이프라인) 정확도 평가 중 최신판을 한 문서로 정리. 각 섹션은 **표 → 한 줄 결론 → 한계** 순서로 구성해 PPT 슬라이드 1개당 섹션 1개로 옮길 수 있게 했다.

## 1. URL 분석(휴리스틱) 정확도 — `docs/system_accuracy_eval.md` (2026-06-21, 최신·최종판)

> 과거 여러 차례 진행된 평가(`run_performance_eval.py`, `run_holdout_eval.py`, `run_crawl_holdout_eval.py`, `run_final_holdout_eval.py`, `run_combined_eval.py`) 중 이 문서가 **튜닝셋 + 독립 홀드아웃을 종합한 최종판**이다(`run_combined_eval.py`는 홀드아웃 로딩 버그로 사용 불가했음).

### 1-1. 튜닝셋 (`performance_test_set.json`, 508건) — 회귀 검증용

| 지표 | 값 |
|---|---|
| 정확도 (Accuracy) | 100.0% |
| 정밀도 (Precision) | 100.0% |
| 재현율 (Recall) | 100.0% |
| FPR / FNR | 0.0% / 0.0% |

**Confusion Matrix**

| True \ Pred | safe | suspicious | danger |
|---|---|---|---|
| safe (182) | 182 | 0 | 0 |
| suspicious (75) | 0 | 75 | 0 |
| danger (251) | 0 | 0 | 251 |

### 1-2. 독립 홀드아웃 (`ctas_holdout.json` 200건 + `safe_holdout.json` 50건 = 250건) — 진짜 일반화 성능

| 지표 | 값 |
|---|---|
| C-TAS 위협탐지율 (suspicious+danger, n=200) | **100.0%** (200/200) |
| 정확히 danger 탐지 (n=200) | 99.5% (199/200, 1건은 suspicious로 약하게 탐지·미탐 아님) |
| Safe 정확 탐지 (n=50) | 100.0% (50/50) |
| Safe FPR | 0.0% (0/50) |
| **SRS 목표 (C-TAS Recall ≥95%)** | **실측 100.0% — PASS** |

**결론**: 한 번도 학습/튜닝에 쓰이지 않은 독립 데이터 250건에서 위협탐지율 100%, 오탐률 0%.

**한계**: `safe_holdout.json`의 10개 도메인이 화이트리스트 구축 참고 도메인과 동일 집합이다. 따라서 Safe FPR 0%는 "화이트리스트 커버리지 안정성"의 증거이지, **미지의 정상 도메인에 대한 일반화 성능을 보증하지는 않는다.** 시스템의 진짜 일반화 성능(미지 위협 탐지력)은 C-TAS recall 100%로만 판단해야 한다.

## 2. 투표 피드백 캘리브레이션 — `docs/sandbox_eval.md` §6~8 (2026-06-21)

> "투표·자동테스트로 누적된 점수가 실제 위험도와 일치하는가"를 실제 DB에 투표를 적재하고 재분석하는 엔드투엔드 실증으로 확인.

### 2-1. 단발 전/후 비교 (§6, §7)

| URL | 시그널 | 투표 전 | danger 투표 10건 후 | 결과 |
|---|---|---|---|---|
| `8.8.8.8/promo` | ip_in_url(35) | 35점, SUSPICIOUS | **70점, DANGER** | 승급 성공 |
| `8.8.4.4/info` | ip_in_url(35) | 35점, SUSPICIOUS | safe 투표 10건→20점, **여전히 SUSPICIOUS** | 톤완화는 됐으나 SAFE 미승격(DC-06) |

### 2-2. 투표 단계별([0,1,3,5,10]건) 점수 추이 (§8)

| 투표 수 | 합성 danger URL(`1.1.1.1`) | 합성 safe URL(`9.9.9.9`) | 실제 C-TAS URL(`y02.o4ys.pw`) |
|---|---|---|---|
| 0건 | 35점 | 35점 | 10점 |
| 1건 | 35점 | 35점 | 10점 |
| 3건 | 55점 | 30점 | 30점 |
| 5건 | 55점 | 30점 | 30점 |
| 10건 | **70점 (DANGER)** | 20점 | 45점 |

**결론**: danger 방향 단조증가, safe 방향 단조감소 — 둘 다 정상. 계단형 곡선(1건→3건 사이 변화 없음)은 `prior_danger_vote_low`(3~9건)/`_high`(≥10건) 임계값 구간 설계이며 버그가 아니다(소규모 어그로 무력화 목적).

**한계**: 검증은 합성 IP URL 3개(`1.1.1.1`/`9.9.9.9`/`8.8.8.8`·`8.8.4.4`)와 실 C-TAS URL 1개(`y02.o4ys.pw`, 블랙리스트 단계를 의도적으로 우회하고 `score_url()` 단독 호출)에 한정된다. 타이포스쿼팅·동형문자 등 다른 시그널 조합 URL이나 실사용자의 자연스러운 투표 분포(가입자/익명 비율, 실제 어뷰징 시나리오)에서의 거동은 이번 검증 범위 밖이다.

## 3. 샌드박스 격리 안전성 — `docs/sandbox_hardening.md` (2026-06-21)

> "가상 서버가 뚫리면 실제 서버나 사용자 기기가 위험해지는가"에 대해 컨테이너 보안 옵션을 실제로 강화하고, 탈출 시도를 직접 실행해 차단 여부를 검증.

### 3-1. 적용된 강화 파라미터

| | 7-A (kasmweb/chromium) | 7-B (Browserless) |
|---|---|---|
| mem_limit | 512m (신규 적용) | 512m (기존) |
| nano_cpus | 0.5코어 (신규 적용) | 0.5코어 (기존) |
| pids_limit | 256 (신규) | 128 (신규) |
| 포트 바인딩 | 127.0.0.1 (이전: 전체 인터페이스 노출 → 수정) | 127.0.0.1 (기존) |
| cap_drop | ALL (신규) | ALL (신규) |
| security_opt | no-new-privileges (신규) | no-new-privileges (신규) |

cap_add 예외 하나도 없이 noVNC 렌더링·CDP 원격디버깅·Playwright 분석 전부 정상 동작 확인(3단계 점진 검증 통과).

### 3-2. 탈출시도 검증 결과 (11/11 차단)

| 시도 | 7-A | 7-B |
|---|---|---|
| 호스트 파일시스템 접근 | 차단 | 차단 |
| 호스트 Docker 소켓 접근 | 차단 | 차단 |
| host.docker.internal SSRF | 차단 | 차단 |
| 프로세스 폭탄(pids_limit) | 차단(요청 768→실제 51) | 차단(요청 384→실제 8) |
| 권한 상승(CapEff) | 차단(CapEff=0) | 차단(CapEff=0) |
| 컨테이너 간 통신(ICC) | 차단(공통 네트워크 테스트) | — |

**결론**: 강화한 6종 탈출 시도 전부 차단 확인. 7-A는 강화 전 메모리/CPU/PID 제한이 전혀 없고 VNC 포트가 전체 노출돼 있던 갭을 7-B 수준으로 끌어올렸다.

**한계**: ICC(컨테이너 간 통신) 테스트는 운영 코드가 세션마다 전용 네트워크를 새로 생성하므로 실제로는 발생하지 않는 상황을 인위적으로 재현한 것이다 — `enable_icc=false` 방어선 자체의 생존만 별도 확인했다. Docker 컨테이너 경계(커널 네임스페이스·cgroup) 너머의 커널 0-day 같은 더 깊은 공격은 이 테스트로 증명할 수 없다 — 모든 컨테이너 기반 격리의 공통 한계.

## 4. 서버 가용성 — `docs/load_test.md` (2026-06-21)

> "실제로 사람들이 얼마나 동시에 접속할 수 있는가"를 가상 사용자별로 다른 IP를 흉내내 레이트리밋을 우회하고 서버 자체 처리 한계를 측정.

### 4-1. `/analyze` 동시 N명 (화이트리스트+휴리스틱 경로, 외부 I/O 없음)

| 동시 사용자 | 2xx 비율 | p50 | p95 |
|---|---|---|---|
| 10명 | 100% | 45.9ms | 75.4ms |
| 30명 | 100% | 236.9ms | 262.8ms |
| 50명 | 100% | 289.9ms | 298.8ms |
| 100명 | 100% | 702.9ms | 710.9ms |

### 4-2. 샌드박스 세마포어 하드캡 검증

| 엔드포인트 | 동시 N | 성공(200) | 503 | 503 응답지연 | 판정 |
|---|---|---|---|---|---|
| `/sandbox/browse` (4슬롯) | 10 | 4 | 6 | p50 43ms / p95 44ms | PASS |
| `/sandbox/browse` (4슬롯) | 100 | 4 | 96 | p50 266ms / p95 274ms | PASS |
| `/sandbox/auto-test` (3슬롯) | 10 | 3 | 7 | p50 38ms / p95 40ms | PASS |
| `/sandbox/auto-test` (3슬롯) | 100 | 3 | 97 | p50 321ms / p95 335ms | PASS |

두 엔드포인트 모두 동시 N과 무관하게 **정확히 슬롯 수만 성공**, 나머지는 1초 이내(n=10일 때 수십ms, n=100일 때 수백ms — n이 늘수록 503 응답도 약간 느려지지만 여전히 빠른 거절 범위)에 503을 받는다 — 설계대로 동작.

### 4-3. 1000명 스트레치 — 서버 다운 (실측 2회 재현)

| 동시 사용자 | 결과 |
|---|---|
| 1000명 | 전체 요청 타임아웃(30초), 이후 서버(`uvicorn --reload`, 단일 워커)가 응답 정지 — **수동 재시작 필요** |

**결론**: `/analyze`는 100명 동시 접속까지 전부 성공·응답시간 1초 이내로 **안전 범위**. 샌드박스 엔드포인트는 설계된 4/3 슬롯 하드캡이 100명 부하에서도 정확히 동작. **1000명에서는 현재 배포 구성(단일 워커 dev 서버)이 실제로 멈춘다는 것을 2회 재현으로 확인** — "100명은 확인된 안전 범위, 1000명은 현재 구성의 실측 한계"라고 구체적으로 답할 수 있다. 프로덕션 배포 시 `uvicorn --workers N` 또는 gunicorn 등 멀티 워커 구성이 필요하다는 근거.

**한계**: 로컬 단일 머신(Windows, Docker Desktop)에서 측정했고, 클라이언트와 서버가 같은 머신이라 네트워크 왕복 지연이 거의 없다 — 실제 인터넷 환경보다 낙관적인 수치일 수 있다. 1000명 레벨은 클라이언트(httpx 커넥션 풀, 단일 프로세스 asyncio) 자체 한계가 서버 한계와 섞일 수 있다.

## 종합 한계 및 후속 과제 (Q&A 대응용)

| 영역 | 한계 | 후속 과제 |
|---|---|---|
| URL 분석 정확도 | Safe holdout 도메인이 화이트리스트 구축 참고 도메인과 중복 — 진짜 일반화는 C-TAS recall(100%)로만 판단 | 화이트리스트 구축에 쓰이지 않은 완전히 새로운 정상 도메인 샘플 확보 |

| 샌드박스 격리 | ICC 테스트는 인위적 재현, 커널 0-day급 공격은 증명 범위 밖 | 정기적 이미지 취약점 스캔(Trivy 등), 커널 패치 최신화 |
