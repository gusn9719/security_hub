## 0. 프로젝트 한눈에 보기

**한 줄 정의**: 의심 문자/URL을 다신호 스코어링 파이프라인으로 분석해
`안전/의심/위험`을 판정하고, 판정이 애매할 때 가상 샌드박스(직접 탐방 또는
AI 자동 테스트)로 사용자가 안전하게 확인해볼 수 있게 하는 Flutter+FastAPI 앱.

**핵심 차별점 — "그냥 차단 도구가 아니다"**:
일반적인 스미싱 차단 앱은 정적 블랙리스트만 있다. 이 프로젝트는 그 위에
**사용자 피드백 학습 순환**을 얹었다:

```
신규 위협 URL 등장 (C-TAS 블랙리스트에 아직 없음)
   → 휴리스틱 점수 30~69점 (SUSPICIOUS, 차단까지는 안 됨)
   → 사용자가 7-A 가상 샌드박스에서 "직접 들어가서" 확인
   → 위험하다고 투표 (danger)
   → url_votes 테이블에 누적
   → 같은 도메인을 다른 사용자가 분석 요청 시
     prior_danger_vote_high/low 시그널이 휴리스틱 점수에 가산
   → 누적되면 DANGER로 승격
```

C-TAS가 아직 못 잡은 신규 피싱을 **집단 투표로 먼저 발견**하고, 그 결과가
**같은 분석 파이프라인의 입력으로 즉시 재반영**되는 폐루프(closed loop) 구조다.
이게 졸업작품에서 제일 강조할 포인트.

다른 설계 원칙(전부 `CLAUDE.md`에 명문화돼 있고 코드로도 강제됨):
- SAFE는 화이트리스트 히트 단 하나의 경로뿐 — 휴리스틱 점수가 아무리 낮아도
  SAFE는 못 나온다 (DC-06). "모르면 의심"이 원칙.
- DANGER는 휴리스틱 단일 시그널로 도달 불가 (최댓값 40점, 임계값 70점) —
  항상 2개 이상 근거가 결합돼야 위험 판정.
- 설명 카드는 LLM이 즉석 생성하지 않는다 — 사전 정의된 딕셔너리에서만 꺼낸다
  (할루시네이션 방지, DC-25). Gemini는 7-B 샌드박스 결과 요약에만 쓴다.

## 2. 분석 파이프라인 상세

오케스트레이터: [`backend/services/analysis_service.py`](../backend/services/analysis_service.py)
(상세: [guide/01_분석파이프라인.md](guide/01_분석파이프라인.md))

Early Return 구조. 동기 블로킹 I/O(SQLite, WHOIS, 단축URL 해제)는 모두
`asyncio.to_thread()`로 분리해 이벤트 루프를 막지 않는다(DC-42).
분석 이력은 `BackgroundTasks`로 응답 후 비동기 저장(DAT-06) — 응답 지연 없음.

### 2.0~1단계 — URL 추출 & 위험 스킴 검사

| 단계 | 모듈 | 동작 |
|---|---|---|
| 0 | `blacklist_service.extract_urls()` | 프로토콜 URL 정규식 + 프로토콜 없는 베어 도메인(`KNOWN_TLDS` 화이트리스트로 오탐 방지) 추출. URL 없으면 즉시 SUSPICIOUS |
| 1 | `url_validator.check_dangerous_scheme()` | `javascript:/data:/vbscript:/file:/blob:` → 즉시 DANGER (ANL-00) |

**한계 (미해결 P0-2)**: `check_dangerous_scheme()`이 `unquote()`나 `strip()`을
하지 않는다. `j%61vascript:alert(1)`(퍼센트 인코딩) 이나 앞에 공백이 붙은
` javascript:alert(1)`은 `urlparse().scheme`이 빈 문자열이 되어 이 단계를
그냥 통과한다. 다만 7단계 `double_encoding` 시그널(+15점)만으로는 DANGER
임계값(70)에 못 미쳐 SUSPICIOUS로 폴백된다 — 위험하지만 안전망 안에 있다.

### 2.2단계 — 단축 URL 해제 + SSRF 방어

모듈: `backend/services/url_expander.py`

- `is_short_url()`: `SHORT_URL_DOMAINS`(bit.ly, forms.gle, goo.gl, naver.me,
  han.gl, url.kr, me2.do, t.co, fb.me, alie.kr, ph.link 등 17개+) 포함 여부.
- `expand_url()`: HEAD 요청(`allow_redirects=False`), 최대 **3-hop**, 루프
  탐지(visited set), Android User-Agent(일부 단축서비스 봇 차단 우회용),
  타임아웃 5초. 실패 시 원본 URL 유지(SUSPICIOUS 폴백 안전망).
- **SSRF 방어**: `_is_safe_to_request()`가 매 hop 전에 `is_private_ip()`로
  사설/루프백/링크로컬 IP 목적지를 차단.

**최근 수정 (P1-1, DC-49, 2026-06-04 발견)**: 신규 함수 `_normalize_ip_host()`
추가. 이전에는 `2130706433`(10진수), `0x7f000001`(16진수), `0177.0.0.1`
(8진수 옥텟) 같은 비표준 IP 표기를 `ipaddress.ip_address()`가 거부해
`is_private_ip()`가 `False`를 반환 → SSRF 우회가 가능했다. 지금은 정규화
후 검사하도록 고쳐 세 가지 우회 패턴 모두 `127.0.0.1`로 정확히 정규화된다.

**미해결 한계**:
1. **DNS rebinding 미방어** — URL 파싱 단계에서 별도 DNS 조회를 하지 않고
   `requests` 내부 동작에 의존. 1차 요청에 정상 IP를 응답하고 2차에 사설
   IP로 바꾸는 공격은 막지 못함(경량 설계 트레이드오프, 해결 안 함).
2. **3-hop 초과 우회 (미해결 P1-2)** — 4단계 이상 리다이렉트 체인이면 3번째
   hop URL을 "최종"으로 잘못 반환한다. 예: `bit.ly→tinyurl→is.gd→실제피싱`
   체인이면 `is.gd` 단계에서 멈춰 실제 피싱 도메인을 분석하지 못함.

### 2.3a/3b단계 — 블랙리스트 3단계 매칭

모듈: `backend/database/blacklist_service.py`

`check_blacklist(urls, hash_only=False)` — **url_hash → domain →
registered_domain** 순서로 매칭(우선순위 = 정확도순):

```
1순위 url_hash 정확일치   — SHA256(이중디코딩+소문자+trailing슬래시 제거 후 URL)
2순위 domain 일치          — hash_only=True면 스킵
3순위 registered_domain 일치 — hash_only=True면 스킵
```

`url_hash` 정규화: `unquote(unquote(url))`(이중 디코딩, OWASP 이중인코딩
우회 방지) → 소문자 → trailing slash 제거 → SHA256.
`load_ctas_csv.py`도 동일 정규화 로직을 공유해야 함(불일치 시 매칭 실패).

**파이프라인에서 3a/3b로 분리한 이유 (DC-46)**: 화이트리스트 보호 *전에*
정확한 피싱 경로(url_hash)는 먼저 막아야 하지만, 도메인 레벨 매칭은
화이트리스트를 통과한 뒤에만 적용해야 단축 URL 서비스가 통째로 막히지 않는다.
그래서 `analysis_service.py`가 `check_blacklist(hash_only=True)`를
화이트리스트 단계 *앞*에서, `check_blacklist(hash_only=False)`를 화이트리스트
단계 *뒤*에서 두 번 호출한다.

**최근 수정 (P0-1, DC-46, 2026-06-04 발견 — 가장 임팩트 큰 버그)**:
C-TAS 블랙리스트에 `forms.gle`, `bit.ly` 등 단축 URL 서비스 도메인 자체가
231건 등재돼 있어서, 2/3순위 도메인 매칭이 그 도메인을 통째로 차단했다.
실제 사례: `"아차사고 공모전 오늘까지 접수 https://forms.gle/aLkBXdQ4..."`
같은 정상 구글폼 공모전 안내가 DANGER로 오판됨. 수정: `_SHORT_URL_PROVIDERS`
frozenset(`forms.gle, goo.gl, bit.ly, naver.me, me2.do, han.gl, url.kr, t.co,
fb.me` 등)을 추가해 이 도메인들은 **url_hash 정확 매칭만** 적용하도록 제한.
정상 경로는 통과, 실제 등재된 피싱 경로는 계속 차단.

**한계**: `_SHORT_URL_PROVIDERS`(blacklist_service.py), `SHORT_URL_DOMAINS`
(url_expander.py) 두 곳을 항상 동기화해야 함 — 신규 단축서비스 추가 시
한쪽만 고치면 누락 위험.

### 2.4단계 — 화이트리스트 + Open Redirect

모듈: `backend/database/whitelist_service.py`

- 매칭 모드: **exact**(`tistory.com` ≠ `evil.tistory.com`), **suffix**
  (`naver.com` 히트 → `login.naver.com`도 포함). `pattern` 모드는 **DC-001로
  제거됨**(`.go.kr` 같은 패턴이 임의 하위도메인을 다 허용해서 오탐 위험 —
  기관별로 exact/suffix 개별 등록으로 전환).
- `_get_suspicious_categories()`: query string에서 `&` 경계로 정확히 파라미터
  매칭(`image_url=` 이 `url=` 로 오탐되지 않게). 카테고리: 리다이렉트
  (`redirect=/goto=/url=` 등), 자격증명(`token=/passwd=` 등), 파일다운로드,
  로그인우회.
- 화이트리스트 히트 + Open Redirect 파라미터 감지 → **DANGER가 아니라
  SUSPICIOUS** (오탐 가능성 고려한 보수적 처리).

**SAFE로 가는 유일한 경로**: 화이트리스트 히트 AND Open Redirect 없음.
이 외에는 무조건 SUSPICIOUS 이상.

### 2.5~6단계 — 도메인 평판 (WHOIS/SSL) + 캐시

모듈: `domain_reputation_service.py` + `reputation_cache_service.py`

- WHOIS 등록일 → `domain_age_days` → 30일 이내면 `new_domain` 플래그.
- SSL 인증서 발급일 → `ssl_issued_days` → 7일 이내면 신선한 SSL.
- 둘 다 신규면 `fresh_infrastructure`(피싱 캠페인 단기 인프라 패턴).
- IP 주소·`SKIP_WHOIS_TLDS`(.local/.test 등)는 조회 자체를 스킵.
- **캐시 TTL 7일** — 같은 등록 도메인 재조회 방지(WHOIS 레이트리밋 완화).
- 모든 외부 조회는 실패해도 `None` 처리(graceful) — SUSPICIOUS 폴백 안전망
  안에서 동작, 서비스 중단 없음.

**한계**: python-whois 응답 형식이 레지스트라별로 들쭉날쭉(날짜 포맷 5종
지원해도 못 잡는 케이스 존재), ccTLD 레이트리밋 가능성, HTTP→HTTPS 실제
리다이렉트 여부는 정적 분석이라 확인 불가.

### 2.7단계 — 휴리스틱 25종 시그널 스코어링

> **표기 정정 (2026-06-21)**: `_WEIGHTS` 딕셔너리를 `len()`으로 직접 확인한
> 결과 25개다. CLAUDE.md를 비롯해 프로젝트 전반에 퍼져 있던 "23종" 표기는
> stale 카운트였음을 이번에 발견하고 정정함(`backend/services/
> analysis_service.py`의 "13 시그널" 구버전 주석도 함께 25로 수정).

모듈: `backend/services/heuristic_scorer.py` — **DANGER_THRESHOLD=70,
SUSPICIOUS_THRESHOLD=30**, 최종 `score = max(0, sum(triggered.values()))`
(DC-06 0클램프 — 음수 시그널이 있어도 SAFE 불가).

★=v0527 신규 5종, ↑=v0527 가중치 상향 2종.

**설계 핵심 — 왜 임계값 70인가**: 가장 강한 단일 시그널(`sandbox_danger_score`)
이 40점이라, 70점에 도달하려면 항상 다른 시그널이 최소 1개 더 결합돼야 한다.
즉 **DANGER 판정은 절대 "단일 근거"로 나오지 않는다** — 설명 가능성과 오탐
방어를 동시에 잡기 위한 설계. 발표에서 "왜 머신러닝을 안 썼나"라고 물으면:
"휴리스틱 + 명시적 가중치는 어떤 판정이든 '왜'를 정확히 보여줄 수 있고(설명
카드), 졸업작품 규모에서 ML은 학습 데이터·검증 인프라 비용이 시그널 설계보다
훨씬 큼" 정도로 답하면 됨.

투표 시그널은 익명/가입자 임계값이 분리(DC-45) — 가입자 1명은 휴대폰 인증을
거쳤으므로 익명 3~3.3명 권위로 환산.

**성능평가 체계 (2026-06-21 실측) — 5단 구조**

휴리스틱은 안전/의심/위험 판정의 일부일 뿐이다(블랙리스트+화이트리스트+
휴리스틱+DC-06 정책의 합). 그래서 "시스템이 얼마나 정확한가"와 "휴리스틱
컴포넌트가 내부적으로 어떻게 동작하는가"를 명확히 나눠서 평가했다 — 이전엔
이 둘을 섞어서 두 번 잘못 보고했었다(아래 (a) 항목의 "정정 기록" 참조).

**(a) 시스템 레벨 3분류 정확도 — 헤드라인 수치
(상세: [docs/system_accuracy_eval.md](system_accuracy_eval.md))**

`AnalysisService.analyze()` 전체 파이프라인(블랙리스트+화이트리스트+
휴리스틱+DC-06)을 실제로 호출해서 측정. 이게 "시스템이 안전/의심/위험을
얼마나 잘 맞히는가"의 진짜 답이다.

| 데이터셋 | 결과 |
|---|---|
| 기존 테스트셋(508건, `run_performance_eval.py`) | **정확도 100.0%**, 3분류 confusion matrix 전부 대각선(오차 0) |
| 독립 홀드아웃(250건, `run_holdout_eval.py`) | danger 탐지율 **100.0%**(200/200), safe FPR **0.0%**(0/50), SRS 목표(≥95%) **PASS** |

**정정 기록**: 처음엔 이 시스템 레벨 질문에 `heuristic_scorer.score_url()`
**단독** 호출 결과(블랙/화이트리스트 우회)를 갖다 써서 "suspicious
Recall=0.000"이라고 두 번이나 잘못 보고했다. 휴리스틱은 의사결정 트리의
한 단계일 뿐이라 단독으로 떼어서 시스템 정확도를 잴 수 없다 — 진짜
시스템 정확도는 위 표(a)이고, 휴리스틱 자체는 아래 (b)에서 별도로 본다.

**(b) 휴리스틱 컴포넌트 단위 분석
(상세: [docs/heuristic_eval.md](heuristic_eval.md))**

`score_url()`을 블랙/화이트리스트 우회하고 직접 호출해 컴포넌트 내부
동작만 본다(시스템 정확도 아님, 시그널 발화 빈도/점수분포/threshold
민감도). `run_performance_eval.py`가 508건 중 80건에서만 호출하고
`domain_evidence`를 빈 값으로 넘겨서 25종 중 10개 시그널(도메인평판 3종+
투표 6종+sandbox 1종)이 데이터로 한 번도 발화 안 시킨다 — 그래서
**`test_heuristic_scorer.py`**(25종 전부 합성입력 양성/음성, **58개 PASS**)로
별도 검증했다. 점수 분포는 danger군 70~130점, suspicious/safe군 거의
0점으로 **이산적/이중모드(bimodal)** — threshold sweep으로 70 설정이
"40~70은 동일 성능, 75부터 급락"하는 지점에 걸쳐 있음을 확인.

**(c) 투표→위험 승급 엔드투엔드 실증**: SUSPICIOUS(35점, `ip_in_url`) →
danger 투표 10건 → 재분석 → **DANGER 승급(70점)**. `prior_danger_vote_high`
가 실제 DB 투표를 거쳐 반영됨을 라이브로 확인(`run_vote_escalation_test.py`).

**(d) 투표→안전 톤완화 엔드투엔드 실증(SAFE 승격 아님)**: 같은 방식으로
SUSPICIOUS(35점) → safe 투표 10건 → 재분석 → **점수만 20점으로 감소**,
**status는 투표 전후 모두 suspicious 유지**(`run_safe_tone_down_test.py`).
"투표가 보수적으로 쌓이면 안전 쪽에도 반영되는가"의 정확한 답: **경고
강도(점수)는 줄어들지만 SAFE로의 승격은 절대 없다** — DC-06(SAFE는
화이트리스트 단일 경로, 절대 안 바뀌는 불변 원칙) 때문. 1종 오류(위험을
안전으로 오판) 비용이 압도적으로 크기 때문에 의도적으로 막아둔 설계.

**(e) 7-B 샌드박스 실데이터 신뢰성**: 위 섹션 3.2 참조
(`docs/sandbox_eval.md`).

**한계**: (a)의 508건은 튜닝에 쓰인 데이터라 100%가 일반화 증거로는
약함(holdout 250건이 더 신뢰할 수 있는 일반화 지표). (b)의 표본
30/50/180건은 작은 편. `domain_evidence=None` 가정은 "첫 등장 URL"
시나리오라 실제보다 보수적인 하한선에 가깝다.

### 2.7-보조 — 타이포스쿼팅 (`domain_similarity.py`)

순수 파이썬 Levenshtein 동적계획법(외부 라이브러리 미사용, O(len(a)×len(b)),
1차원 롤링 배열로 메모리 절약). `detect_typosquat(registered_domain,
max_distance=2)`:
- 정확 일치면 미발동(정상). 표적 도메인 35개(포털/금융/커머스/글로벌/정부
  — `naver.com, kakao.com, toss.im, hometax.go.kr, coupang.com, paypal.com` 등)
  와 거리 계산.
- 길이 차이가 `max_distance`보다 크면 가지치기(스킵), 길이 < 6인 표적은
  제외(예: `sk.com`은 짧아서 거리 1로도 무관한 도메인과 오탐 가능 — 안전장치).
- 거리 1 발견 시 조기 종료(더 짧은 거리는 있을 수 없음).

**한계**: 표적 도메인 35개로 제한(신규 핀테크 등 미포함), `naver-event.com`
같은 부분 변형은 거리 2 이내면 잡히지만 TLD 변형(`naver.co.kr` vs
`naver.com`, 거리 5)은 못 잡음.

### 2.8단계 — 설명 카드 (`explanation_service.py`)

`EXPLANATION_DICT`(30개+ 카드, `{icon, title, desc}`)에서 발화된 시그널을
점수 높은 순으로 매핑. **Gemini가 즉석 생성하지 않음**(DC-25, 할루시네이션
방지). `build_safe_cards`/`build_blacklist_cards`(category별 힌트: 공공기관/
택배/금융)도 동일하게 딕셔너리 기반.

### 3.1 7-A 직접 탐방

**아키텍처**: `Flutter WebView → (HTTP 평문) → FastAPI SSL-strip 프록시
(127.0.0.1:동적포트) → KasmVNC HTTPS(자체서명 인증서)`. Android WebView가

**Pre-warm pool**: 서버 부팅 시 컨테이너 2개를 미리 만들어두고, 요청이 오면

풀이 비었으면 온디맨드 생성(pool miss, 30~80초 — 콜드스타트는 어쩔 수 없음).

**SSL-strip 프록시** (`_proxy_handle`): HTTP 헤더 파싱 → Authorization
Basic 헤더 자동 주입(VNC_PW는 컨테이너마다 `uuid4().hex`로 랜덤, DC-27) →
타겟에 TLS로 재연결 → WebSocket이면 바이트 양방향 릴레이, HTML이면 절대경로
재작성(`src="/"` → `/novnc/`) + CSP 메타태그 제거 + `fetch`/`XHR` 패치
스크립트 주입(상대경로를 `/sandbox/browse/{id}/novnc/...`로 재작성).

**noVNC WebSocket 프록시** (`routers/sandbox.py`의 `_novnc_ws_proxy`):
`websockets.connect(wss://127.0.0.1:{kasm_port}/, ...)`로 직접 연결,
`ping_interval=30`(Cloudflare Tunnel 100초 타임아웃 방어), `max_size=10MB`.

**CDP 위협 자동 차단 (DC-34)** — ⚠️ **CLAUDE.md 구버전 기록은 "미구현"이라고

1. 컨테이너 내부에서 Python CDP 모니터 스크립트를 `exec_run(stream=True,

5. **프론트엔드** `sandbox_browse_screen.dart`가 시작 직후부터 **2초 간격**

### 3.3 Semaphore 동시성 — 알려진 미해결 이슈

`routers/sandbox.py`:
```python
if _AUTO_SEM._value == 0:          # 비공개 API, 사전 검사
    raise HTTPException(503, ...)
async with _AUTO_SEM:              # 검사~획득 사이 경합 가능
```
두 코루틴이 동시에 "마지막 슬롯이 비어있음"을 보고 둘 다 `async with`로
진입을 시도하면, asyncio.Semaphore의 설계상 한쪽은 즉시 503이 아니라 대기
큐에 들어간다(503 즉시 거부 의도와 미세하게 어긋남). 근본 해결은
`asyncio.Semaphore`가 "non-blocking try-acquire"를 공식 지원하지 않아서
구조적으로 어렵다 — 졸업작품 범위상 보류 상태(기능적 영향은 미미: 최악의
경우 동시 요청자가 503 대신 짧게 대기).

## 4. 투표 시스템 + 어그로 방어 5중 (상세: [guide/03_투표인증.md](guide/03_투표인증.md))

모듈: `backend/database/vote_service.py`

`save_vote()`: vote가 `safe/danger/spam`이면 `INSERT OR IGNORE`로
부분 UNIQUE 인덱스 슬롯을 점유. `unsure`는 **DB에 저장하지 않음**(슬롯
미점유 — 나중에 진짜 의견으로 재투표할 수 있게 비워둠).

```sql
UNIQUE (device_uuid, registered_domain)
  WHERE vote IN ('safe', 'danger', 'spam')   -- unsure 제외
```

`get_vote_counts()`: `user_id IS NULL`이면 `anon`, 아니면 `user` 버킷으로
분리 집계(`anon_safe/anon_danger/anon_spam`, `user_safe/user_danger/
user_spam`, 합계 `safe/danger/spam/total`).

### 어그로 방어 5중

| Layer | 구현 | 위치 | 효과 |
|---|---|---|---|
| 1. DB | 부분 UNIQUE 인덱스 | `vote_service.py` + 스키마 | 1기기 1표(도메인당) |
| 2. Logic | 우세방향 가드 | `heuristic_scorer.py` | danger>safe 등 명확한 우세일 때만 시그널 발동, 동률이면 무발동 |
| 3. UX | 체류시간 | `browse_service.py` `_auto_terminate` 5분 | **불완전** — 강제 체류 아니라 자동정리 타임아웃일 뿐. 30초 최소체류 강제는 미구현 |
| 4. Score | 점수상한 | `heuristic_scorer.py` | `prior_danger_vote_high` 최대 +35, 단독 DANGER 불가(70 미달) |
| 5. Auth | 카카오 인증 | `auth.py`/`user_service.py` | 가입자 1명=익명 3~3.3명 권위(임계값 anon≥10 vs user≥3) |

**Layer 3는 정직하게 한계로 인정해야 함** — CLAUDE.md에도 "UX 체류시간"
이라 써있지만 실제로는 5분 타임아웃(컨테이너 자동 정리)일 뿐, 30초 미만
체류 시 투표를 막는 강제 로직은 없다. 질문 들어오면 "현재는 소프트
가드(Layer 1·2·4·5)만으로 충분히 비용 비대칭을 만들고 있고, 강제 체류
타이머는 졸업작품 범위에서 보류했다"고 답하는 게 정확함.

**투표→재분석 승급 엔드투엔드 실증 (2026-06-21, 상세: [docs/sandbox_eval.md](sandbox_eval.md) §6)**:
"의심으로 넘어간 게 어떻게 처리되는지"를 코드 설명이 아니라 실제
`analysis_service.analyze()` + `vote_service.save_vote()` 호출로 직접
보여줬다. 테스트 URL `http://8.8.8.8/promo`(ip_in_url 단일 시그널,
baseline 35점)를 분석 → **suspicious** 확인 → 서로 다른 device_uuid로
danger 투표 10건 적재 → 같은 URL 재분석 → **danger**로 승급(35+35=70).
`prior_danger_vote_high`가 실제 DB 투표를 거쳐 살아있는 파이프라인에서
정확히 반영됨을 처음으로 실데이터로 확인(이전엔 `score_url()`에 합성
`vote_counts`를 직접 주입하는 단위테스트만 있었음). 테스트 투표는 종료 후
자동 삭제해 운영 DB에 영향 없음.

**관찰**: 이 테스트로 드러난 부수적 사실 — `AnalyzeResponse`는 `status`/
`title`/`description`/`cards`만 반환하고 휴리스틱 점수 숫자나 `triggered`
딕셔너리를 노출하지 않는다. 사용자는 "왜 35점에서 70점으로 올랐는지"를
점수로 볼 수 없고 카드 텍스트로만 추론해야 한다(설명카드 투명성 공백,
별도 보류 항목).

## 5. 인증 — 카카오 로그인 + JWT (AUTH-01~03) (상세: [guide/03_투표인증.md](guide/03_투표인증.md))

### 백엔드 흐름

```
Flutter (kakao_flutter_sdk_user, talk→web 폴백)
  → POST /auth/kakao { access_token }
  → kakao_service.fetch_user_info()  — kapi.kakao.com/v2/user/me 직접 검증, 5초 타임아웃
  → user_service.upsert_by_kakao_id() — 신규면 INSERT, 기존이면 nickname/email/last_login_at UPDATE
  → jwt_service.issue_token(user_id) — HS256, sub=user_id, exp=720시간(30일)
  → 응답 { access_token, user: {id, kakao_id, nickname, ...} }
```

- **JWT 시크릿 32자 미만 거부**(RFC 7518 §3.2) — `jwt_service.py`가
  앱 시작 시점에 검증, 짧으면 `ValueError`.
- **PII 마스킹**: `user_service._mask_kakao_id()`가 로그에 `1234******`
  형태로만 남김. kapi 응답 body 자체도 로그에 안 남김.
- `OptionalAuthMiddleware`: 토큰 없으면 익명 통과(`request.state.user_id =
  None`), **무효 토큰(만료/서명오류)은 무조건 401** — silent pass-through로
  두면 클라이언트가 죽은 토큰을 계속 들고 있어서 익명 가중치로 계속 동작하게
  되므로, 명시적으로 401을 줘서 클라이언트가 즉시 폐기하도록 강제.

### 프론트엔드 흐름

`AuthService`(`frontend/lib/services/auth_service.dart`): talk 로그인
실패 시(사용자 취소가 아닌 진짜 오류) web 로그인으로 폴백 → 백엔드 교환 →
JWT+프로필을 SharedPreferences에 영속화. `ApiService._headers()`가 매
요청마다 JWT 존재 시 자동으로 `Authorization: Bearer` 부착. 401 수신 시
`AuthService.clearLocalAuth()`로 즉시 폐기 후 익명 모드 전환(자동 재로그인은
시도 안 함 — 카카오 SDK가 사용자 인터랙션을 요구하기 때문).

`LoginScreen`: 카카오 로그인 버튼(공식 색상 #FEE500) + "비회원으로 시작"
버튼 — 비회원 선택도 `markAnonymous()`로 영속화해 다음 실행에 또 묻지 않음.

## 6. 보안 미들웨어 체인 (상세: [guide/04_미들웨어DB.md](guide/04_미들웨어DB.md))

`backend/main.py`. 실행 순서(바깥→안쪽): **CORS → SecurityHeaders →
RateLimit → DeviceUUID → OptionalAuth → BlockDangerousMethods → 핸들러**

| 미들웨어 | 역할 | 핵심 |
|---|---|---|
| CORS | `allow_origins=["*"]` | TRACE/CONNECT/TRACK은 `allow_methods`에서 명시적으로 제외 |
| SecurityHeaders | `X-Content-Type-Options: nosniff`, `X-XSS-Protection`, `Referrer-Policy: no-referrer`, `Cache-Control: no-store`(NF-12), CSP | noVNC 정적자산(`.js/.css/.png` 등)만 캐시 허용. `/test-phishing`은 `unsafe-inline`+`blob:` CSP(테스트 페이지 특성상 필요) |
| RateLimit | IP 기반(NF-24) | `/analyze` 10/min, `/sandbox/browse`·`/auto-test` 5/min, `/sandbox/votes` 20/min, `/auth/kakao` 5/min. IP 추출: `cf-connecting-ip`→`x-real-ip`→`x-forwarded-for`→`client.host` 순 |
| DeviceUUID | `X-Device-UUID` 강제(NF-30) | UUID v4 검증, 없으면 401, 형식오류면 400. 제외경로: `/docs` 등 + 정규식 `^/sandbox/browse/[^/]+/novnc(?:/|$)`(DC-43 — 이전엔 `"/novnc" in path` 단순 부분일치라 `/api/novnc-test` 같은 경로로 우회 가능했음) |
| OptionalAuth | JWT 선택적 검증 | 섹션 5 참조 |
| BlockDangerousMethods | TRACE/CONNECT/TRACK → 405 | XST 공격, 프록시 터널링 악용 차단 |

**전역 예외 핸들러**: 클라이언트에는 일반 메시지만(`서버 내부 오류가
발생했습니다`), 스택 트레이스는 서버 로그에만.

**RateLimit 메모리 누수 수정(DC-44)**: 윈도우 밖 타임스탬프만 정리하고
빈 리스트 키를 `dict`에 남겨두면 IP×엔드포인트 조합 수만큼 키가 영구
누적된다 — `.get(key, ())`로 읽어서 비어있으면 키 자체를 안 만들도록 수정.

**미해결 한계**: `x-forwarded-for`를 무조건 신뢰 — Cloudflare Tunnel 없는
환경에서는 클라이언트가 이 헤더를 위조해 레이트리밋을 우회할 수 있음
(CF Tunnel 운영 시에만 `cf-connecting-ip`가 신뢰할 수 있는 값).

## 7. DB 스키마 (상세: [guide/04_미들웨어DB.md](guide/04_미들웨어DB.md))

SQLite `backend/security_hub.db`, WAL 모드 + `busy_timeout=5000` +
`synchronous=NORMAL`. **읽기/쓰기 커넥션 분리(DC-17)**: 런타임 요청은
`get_ro_connection()`(`mode=ro` URI — 의도하지 않은 쓰기 시 즉시
`OperationalError`), 스크립트/초기화만 `get_rw_connection()`.

| 테이블 | 핵심 키/제약 | 역할 |
|---|---|---|
| `blacklist` | `url_hash` UNIQUE, `domain`/`registered_domain` 인덱스 | C-TAS 피싱 URL |
| `whitelist` | `domain` UNIQUE, `match_mode`(exact/suffix) | 안전 도메인. `pattern` 모드는 DC-001로 제거 |
| `domain_reputation_cache` | `registered_domain` UNIQUE, `expires_at` | WHOIS/SSL 캐시 TTL 7일 |
| `url_votes` | `session_id` UNIQUE, 부분 `UNIQUE(device_uuid, registered_domain) WHERE vote IN (...)`, `user_id NULL`(AUTH-01) | 7-A 투표. `CHECK(vote IN ('safe','danger','spam','unsure'))` |
| `sandbox_results` | `session_id` PK(DC-33로 `url_hash UNIQUE` 제거) | 7-A/7-B 결과, TTL 24h |
| `analysis_history` | `url_hash`, `analyzed_at` 인덱스 | 분석 이력(BackgroundTasks 비동기 INSERT, DAT-06) |
| `users` | `kakao_id` UNIQUE(AUTH-01) | 카카오 가입자. FK 미설정(SQLite `foreign_keys=OFF`, 탈퇴 흐름은 범위 밖) |

**마이그레이션 전략**: `PRAGMA table_info`로 컬럼 존재를 확인하고 없으면
조건부 `ALTER TABLE`로 추가 — 기존 DB를 깨지 않고 스키마를 진화시킴
(예: `whitelist.match_mode/risk_level/note/registered_domain`,
`url_votes.user_id`).

### 7-A WebView 통합의 디테일

`sandbox_browse_screen.dart`가 JS 주입으로 처리하는 것들:
- **WebSocket 재작성**: `window.WebSocket`을 `Proxy`로 가로채 `wss://임의호스트`
  를 `ws://서버/sandbox/browse/{id}/novnc`로 재작성(자체서명 인증서 문제 회피).
- **절대경로 fetch/XHR 재작성**: `/api/...` 같은 절대경로를
  `pathBase + 그경로`로 변환.
- **세션 종료 감지**: 8초 폴링(`#noVNC_status` 텍스트에 "disconnect" 포함
  검사) + `MutationObserver`(동적 추가 노드에 "session"+"expired/timeout"
  텍스트 검사) 2중 감지.
- **터치 오버레이**: noVNC 기본 터치 핸들러가 Pointer Lock으로 캔버스를
  독점하는 문제를 막기 위해 투명 `<div>`를 캔버스 위에 깔고 탭/스와이프를
  자체 판정(10px 이하 이동=탭→마우스클릭, 그 이상=스와이프→WheelEvent).
- **한글 IME**: `compositionend` 이벤트를 noVNC의 키보드 입력 핸들러용
  `InputEvent`로 변환 중계.
- **투표 모달 await 수정(P0-6)**: 예전엔 `submitVote()`를 fire-and-forget
  으로 호출해서 네트워크 오류 시 사용자는 "저장됨"으로 봤지만 DB엔 없었음.
  지금은 `await`로 결과를 확인하고, 실패 시 `_voteDone`을 갱신하지 않아
  다음 종료 시 재시도 가능하게 함.

실행하면 거의 전 항목이 FAIL/측정불가로 나온다. 원인은 서버가 아니라
스크립트 자체의 설계 결함 — `run_perf_whitelist()` 한 함수가 워밍업 10회 +
측정 20회 = 30회를 같은 IP·같은 엔드포인트(`/analyze`)로 연속 호출하는데,
`/analyze`의 레이트리밋이 10회/60초라서 워밍업 단계에서 이미 쿼터를 다
써버린다. 그래서 이전 `qa_benchmark.md`가 항상 "측정 대기" 상태였던 것으로
보인다. 이번엔 카테고리 사이 65초 대기 + 호출량을 쿼터 이하로 낮춘 경량
스크립트로 재측정해 실제 수치를 얻었다. (이건 역설적으로 레이트리밋이
설계대로 엄격하게 작동한다는 증거이기도 함 — 발표에서 좋은 소재.)

## 12. 예상 질문 Q&A (상세: [guide/07_예상질문QA.md](guide/07_예상질문QA.md))

### 분석 파이프라인

**Q. 왜 머신러닝 모델을 안 쓰고 휴리스틱 가중합을 썼나?**
A. (1) 설명 가능성 — 휴리스틱은 "왜 위험한가"를 시그널 단위로 정확히
보여줄 수 있어 설명 카드 생성이 자연스럽다. ML은 블랙박스라 신뢰 구축이
더 어렵다. (2) 데이터 — 졸업작품 규모에서 학습/검증용 레이블 데이터를
충분히 모으기 어렵고, 휴리스틱은 KrCERT/APWG/OWASP/RFC 등 1차 출처 근거로
즉시 검증 가능하다. (3) 투표 피드백 루프와의 결합이 쉽다 — 시그널 가중치
형태라 `prior_danger_vote_*`를 그냥 더하면 끝이지만 ML이면 재학습 파이프라인이
필요하다.

**Q. DANGER 임계값이 왜 70인가?**
A. 가장 강한 단일 시그널(`sandbox_danger_score`, 능동탐지)이 40점이라
70에 도달하려면 항상 다른 시그널이 최소 하나 더 필요하다 — 단일 근거로
DANGER가 나오지 않게 만든 의도적 설계. (이전 버전은 더 낮았는데, 휴리스틱
단독 차단의 오탐 부담을 줄이고 샌드박스 체험 유도 폭을 넓히려고 v0513에
70으로 상향했다.)

**Q. SAFE는 왜 화이트리스트 히트로만 나오나? 휴리스틱 점수가 0이면 안전한
거 아닌가?**
A. "모르면 안전"이 아니라 "모르면 의심"이 원칙(DC-06). 휴리스틱이 모든
위험 패턴을 다 알 수는 없으므로, 점수가 낮다는 것은 "알려진 위험 신호가
없다"는 뜻일 뿐 "안전이 확인됐다"는 뜻이 아니다. 화이트리스트는 직접
검증된 공식 도메인 목록이라 그 경우만 SAFE로 인정한다. 1종 오류(위험을
안전으로 오판)를 구조적으로 최소화하는 설계.

**Q. 휴리스틱 25종 시그널 중 가장 자신 있는 거 하나 설명해보라.**
A. `typosquat_levenshtein` — `naverr.com`처럼 1~2글자 변형된 도메인을
순수 Levenshtein 거리(동적계획법, 외부 라이브러리 없이 직접 구현)로
탐지. 35개 표적 도메인(포털/금융/공공기관)과 거리 1~2 이내면 발화,
길이<6 도메인은 오탐 방지로 제외. 외부 의존성 0, O(n×m) 시간복잡도로
가볍다.

**Q. "시스템 정확도"와 "휴리스틱 컴포넌트 분석"은 뭐가 다른가?**
A. 시스템 정확도(`docs/system_accuracy_eval.md`)는 `AnalysisService.
analyze()` 전체(블랙리스트+화이트리스트+휴리스틱+DC-06)를 실제로 호출해서
"안전/의심/위험을 시스템이 얼마나 잘 맞히는가"를 잰다 — 508건 100%,
독립 holdout 250건도 danger recall 100%/safe FPR 0%. 휴리스틱 컴포넌트
분석(`docs/heuristic_eval.md`)은 그중 7단계(`score_url()`) 하나만 떼어서
"이 컴포넌트 혼자 내부적으로 어떻게 동작하는가"(시그널 발화 빈도, 점수
분포, threshold 민감도)를 본다. 이 둘을 섞으면 안 된다 — 실제로 한 번
섞어서 "의심 클래스 재현율 0%"라는 틀린 결론을 냈었다(아래에서 설명).

**Q. 그 "틀린 결론" 얘기 좀 더 해달라.**
A. 처음에 `score_url()` 단독 결과(블랙/화이트리스트 우회)를 갖고 "시스템이
의심을 얼마나 잘 맞히는가"를 채점하려고 했다. 휴리스틱은 트리의 한
단계일 뿐이라 단독으로는 시스템 정확도를 잴 수 없는데 그걸 잘 모르고
시도해서, raw 내부 판정만 보고 "suspicious 재현율 0%"라고 잘못 보고했다.
DC-06 정책 보정을 빼놓은 게 원인이었다. 이후 진짜 시스템 정확도는 실제
`analyze()` 파이프라인을 호출하는 기존 스크립트(`run_performance_eval.py`/
`run_holdout_eval.py`)로 다시 측정했고, 휴리스틱 단독 분석은 "컴포넌트
내부 동작"으로 명확히 재라벨링했다.

**Q. 휴리스틱 "단독" 컴포넌트 분석이 왜 따로 필요한가?**
A. 시스템 정확도는 블랙리스트/화이트리스트가 먼저 가로채는 구조라
휴리스틱이 실제로 호출되는 케이스가 508건 중 80건뿐이고, 그마저도
도메인평판/투표/샌드박스 입력을 빈 값으로 넘겨서 25종 중 10개 시그널이
한 번도 발화하지 않는다. "시스템이 잘 맞는다"는 결론과 "휴리스틱 점수
체계 자체의 내부 동작이 건전하다"는 결론은 다른 질문이라, 휴리스틱만
분리해서 점수분포/threshold/시그널 기여도를 따로 봤다.

**Q. 10개 시그널이 "사각지대"였다는 게 정확히 무슨 뜻인가?**
A. 컴포넌트 분석 스크립트가 `score_url()`을 호출할 때 도메인평판
(`domain_evidence`)·투표(`vote_counts`)·샌드박스(`sandbox_score`) 인자를
다 빈 값으로 넘겼다. 그래서 그 인자에 의존하는 10개 시그널
(`new_domain`/`whois_no_record`/`fresh_infrastructure`/투표 6종/
`sandbox_danger_score`)은 데이터 기반 평가로는 한 번도 검증된 적이
없었다 — 실제로 잘 동작하는지 "찍어서" 안 게 아니라 "안 거쳐봐서"
몰랐던 것. 합성 입력을 직접 넣는 단위테스트(`test_heuristic_scorer.py`,
58개 PASS)로 메웠다.

**Q. 투표가 위험 쪽으로만 가는 게 아니라 안전 쪽으로도 보수적으로
반영되는가? 왜 화이트리스트 자동 추가 같은 더 적극적인 안전 승급은
안 했나?**
A. 한다 — 단, "승격"이 아니라 "톤 완화"다. `prior_safe_vote_high`(-15)/
`prior_safe_vote_low`(-5)가 safe 투표 누적 시 경고 점수를 실제로 깎는다
(실증: 35점→20점, `docs/sandbox_eval.md` §7). 하지만 점수가 얼마나
떨어지든 화이트리스트가 없는 한 최종 판정은 항상 SUSPICIOUS로 유지된다
— 같은 테스트에서 투표 전후 status가 둘 다 suspicious였다. 화이트리스트
자동추가를 안 한 이유는 의도적이다: 위험을 안전으로 오판하는 1종 오류의
비용이, 안전한 걸 의심으로 잘못 두는 비용보다 압도적으로 크기 때문에
(DC-06, 절대 안 바뀌는 아키텍처 불변 원칙) — 화이트리스트는 수동 검증된
도메인만 들어가는 별도 경로로 분리해뒀다.

### 샌드박스

**Q. 7-A와 7-B의 차이는 뭔가, 왜 둘 다 필요한가?**
A. 7-A는 사용자가 직접 가상 브라우저에 들어가서 보고 판단 — 사람의 직관이
중요한 케이스(디자인이 그럴듯하게 사칭한 경우 등)에 강하다. 7-B는
Playwright로 가짜 정보를 자동 주입해서 폼 동작·리다이렉트를 기계적으로
검사 — 사람 개입 없이 24시간 캐시로 빠르게 재사용 가능. 7-A의 투표
결과(danger/safe)는 다음 분석의 `prior_*_vote` 시그널로, 7-B의
sandbox_score는 `sandbox_danger_score` 시그널로 각각 휴리스틱에 피드백된다.

**Q. kasmweb 컨테이너 보안은 어떻게 보장하나?**
A. 컨테이너마다 랜덤 UUID 비밀번호(`VNC_PW`), `127.0.0.1` 바인딩(외부
직접 노출 안 됨, 백엔드 프록시로만 접근), 격리된 Docker 네트워크
(컨테이너 간 통신 차단), CDP watchdog으로 실시간 navigation 감시 +
블랙리스트 재매칭 + 다운로드 시도 즉시 차단.

**Q. DC-34(CDP 자동 위협 차단)는 구현됐나?**
A. 네, 구현 완료다(2026-06-20 코드로 직접 재확인하고 과거 stale 문서를
정정했다). 백엔드는 CDP `Page.frameNavigated`/`Page.downloadWillBegin`을
실시간 감시해 블랙리스트 히트나 다운로드 시도 시 즉시 세션을 종료하고
스크린샷을 캐싱한다. 프론트엔드는 2초마다 상태를 폴링해서 차단 오버레이를
띄운다.

### 인증/투표

**Q. 카카오 로그인이 왜 필요한가, 단순 익명 UUID로는 부족한가?**
A. 익명 UUID는 0원으로 무한 생성 가능(에뮬레이터 재설치 등)해서 투표
시스템을 조작하기 쉽다. 카카오는 실명확인은 아니지만 휴대폰 SMS 본인인증을
거치므로, 한 사람이 동원할 수 있는 회선 수가 1~2개로 제한돼 어그로 비용이
한 자릿수 이상 증가한다. 가입자 1명 ≈ 익명 3~3.3명 권위로 환산해서
임계값을 분리했다.

**Q. 투표 어그로 방어가 완벽한가?**
A. 아니다, 정직하게 5중 중 Layer 3(체류시간 강제)는 소프트 가드일 뿐
실제 강제 타이머는 없다는 한계가 있다. 나머지 4중(DB UNIQUE, 우세방향
로직, 점수상한, 카카오 인증)으로 비용 비대칭을 만들어 현실적인 수준의
방어를 하고 있다.

### 보안/한계 전반

**Q. 가장 심각하다고 생각하는 미해결 이슈는?**
A. P1-4(RateLimit이 `x-forwarded-for`를 무조건 신뢰)다. Cloudflare Tunnel
환경에서는 `cf-connecting-ip`가 우선이라 안전하지만, 그게 없는 환경에서는
클라이언트가 헤더를 위조해 레이트리밋을 무력화할 수 있다. 다음 우선순위로
고칠 항목으로 꼽고 있다.

**Q. 이번에 발견한 가장 흥미로운 버그는?**
A. 우리 자체 NF 벤치마크 스크립트(`run_qa_benchmark.py`)가 자기가 만든
호출량 때문에 우리 레이트리밋에 걸려서 측정 자체가 실패하던 문제다. 역설적으로
레이트리밋이 의도대로 엄격하게 작동한다는 증거였다.
